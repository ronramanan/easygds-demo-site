# Autocomplete & Ranking Improvements — Implementation Guide

This document describes the custom ranking, sorting, and search enhancements applied to the frontend autocomplete system in `scripts.js`. These changes span **three foundational layers**: static ranking data, scoring/grouping logic, and smart query augmentation.

---

## Layer 1: Static Ranking Data Structures

Four data structures power the ranking system. These are defined as constants near the top of the autocomplete logic.

### 1.1 `AIRPORT_TRAFFIC_RANK`

A flat object mapping **IATA codes to rank position** based on global passenger traffic volume.

```javascript
const AIRPORT_TRAFFIC_RANK = {
  "ATL": 1, "DXB": 2, "DFW": 3, "LHR": 4, // ...~200 entries
};
```

- **Purpose**: Sort airports by real-world traffic instead of alphabetically.
- **Lookup helper**: `getAirportTrafficScore(code)` returns the rank, or `9999` for unlisted airports.

### 1.2 `DESTINATION_POPULARITY_RANK`

A large object mapping **`"normalized_name:country_code" → rank`** for ~2,000 global destinations, generated by a build script (`scripts/generate-destination-ranks.js`) using GeoNames population data + Wikidata sitelinks as a composite tourism relevance score.

```javascript
const DESTINATION_POPULARITY_RANK = {
  "london:gb": 1, "paris:fr": 2, "new york city:us": 3, // ...~2000 entries
};
```

- **Build pipeline**: GeoNames cities (population) + Wikidata SPARQL (sitelinks as tourism proxy) + manual `tourism-overrides.json` (~91 small mega-tourism towns like Santorini, Machu Picchu).
- Tourism-override entries receive a population-percentile floor of 50% and +10 sitelinks bonus.

### 1.3 `COUNTRY_TOURISM_RANK`

Top ~50 countries ranked by UNWTO international arrivals. Used as a **fallback** when a city isn't found in `DESTINATION_POPULARITY_RANK`.

```javascript
const COUNTRY_TOURISM_RANK = {
  "fr": 1, "es": 2, "us": 3, "cn": 4, // ...~50 entries
};
```

### 1.4 `DEST_NAME_ALIASES`

Maps common English names to the local/ASCII names used in the API data, resolving mismatches where GeoNames uses local-language names.

```javascript
const DEST_NAME_ALIASES = {
  "seville": "sevilla", "cologne": "koeln", "munich": "muenchen",
  "prague": "praha", "rome": "roma", // ...etc
};
```

## Layer 2: Scoring & Sorting Logic

### 2.1 `getDestinationPopularityScore(result)`

The core ranking function. Uses a **multi-attempt cascading lookup** to match an API result to a rank:

| Priority | Strategy | Example |
|---|---|---|
| 1 | Direct match: `normalize(city) + ":" + country` | `"paris:fr"` → rank 2 |
| 2 | Alias match: map through `DEST_NAME_ALIASES` first | `"seville"` → `"sevilla:es"` |
| 3 | Toggle `" city"` suffix (add/remove) | `"new york city:us"` ↔ `"new york:us"` |
| 4 | Extract city from display name (strip comma/paren) | `"Barcelona, Spain"` → `"barcelona"` |
| 5 | Alias of extracted name | Same alias logic on extracted name |
| Fallback | `COUNTRY_TOURISM_RANK[country] + 2000` | Unknown Spanish city → 2002 |
| Default | No match at all | → 9999 |

**Key fields used from API results:**
- `city` = `p.name` (short place name)
- `country` = `p.location?.country_code`

### 2.2 Result Grouping — The `groupResults()` Dispatcher

Routes results to type-specific grouping functions based on the input's `data-type` attribute:

| `data-type` | Grouping Function | Behavior |
|---|---|---|
| `airport_code` | `groupAirportResults()` | Groups by city, sorts by traffic rank |
| `any` | `groupHotelDestResults()` | Separates destinations/airports/hotels, sorts each |
| `hotel` | `groupTransferHotelResults()` | Flat list sorted by stars |
| `tour_region` | `groupTourResults()` | Separates destinations from tours, sorts by popularity |

### 2.3 Sort Comparators

There are **7 sort comparators** across the system:

#### Airports — within a multi-airport city group
1. Traffic rank ascending (busiest first)
2. IATA code alphabetical (tiebreaker)

```javascript
const sortedAirports = group.sort((a, b) => {
    const aRank = getAirportTrafficScore(a.code);
    const bRank = getAirportTrafficScore(b.code);
    if (aRank !== bRank) return aRank - bRank;
    return (a.code || '').localeCompare(b.code || '');
});
```

#### City groups — across groups
1. Best (lowest) traffic rank among the group's airports

```javascript
sections.sort((a, b) => {
    const bestRank = (section) => Math.min(
        ...section.items.map(i => getAirportTrafficScore(i.code))
    );
    return bestRank(a) - bestRank(b);
});
```

#### Destinations (hotel search & tour search) — 4-tier sort
1. **Exact match** → **starts-with** → **other** (name relevance)
2. **Destination popularity rank** ascending (most popular first)
3. **Type hierarchy**: `country` → `administrative_area_level_3` → `administrative_area_level_4` → other
4. **Alphabetical** name (final tiebreaker)

```javascript
destinations.sort((a, b) => {
    const aName = (a.city || a.name || '').toLowerCase();
    const bName = (b.city || b.name || '').toLowerCase();
    const aExact = aName === query ? 0 : (aName.startsWith(query) ? 1 : 2);
    const bExact = bName === query ? 0 : (bName.startsWith(query) ? 1 : 2);
    if (aExact !== bExact) return aExact - bExact;

    const aPop = getDestinationPopularityScore(a);
    const bPop = getDestinationPopularityScore(b);
    if (aPop !== bPop) return aPop - bPop;

    const typeOrder = {
        'country': 0, 'administrative_area_level_3': 1, 'administrative_area_level_4': 2
    };
    const aOrder = typeOrder[a.rawType] ?? 3;
    const bOrder = typeOrder[b.rawType] ?? 3;
    if (aOrder !== bOrder) return aOrder - bOrder;

    return (a.name || '').localeCompare(b.name || '');
});
```

#### Hotels
1. Stars descending (5-star first)
2. Name alphabetical (tiebreaker)

```javascript
hotels.sort((a, b) =>
    (b.stars || 0) - (a.stars || 0) || (a.name || '').localeCompare(b.name || '')
);
```

## Layer 3: Smart Query Augmentation

### 3.1 Popular Destination Prefix Injection

**Problem**: Typing a short prefix like `"bar"` returns random results because the API doesn't know about destination popularity.

**Solution**: For queries matching a prefix of a top-200 destination name, fire **parallel supplementary API calls** for the full destination names alongside the primary query.

#### Constants

```javascript
const POPULAR_DEST_THRESHOLD = 200;   // Only top-200 destinations
const MAX_SUPPLEMENTARY_CALLS = 3;    // Cap parallel calls
```

#### Core Function: `findPopularPrefixMatches(query)`

```javascript
function findPopularPrefixMatches(query) {
    const q = normalizeDestName(query);
    if (!q) return [];

    const matches = [];
    for (const [key, rank] of Object.entries(DESTINATION_POPULARITY_RANK)) {
        if (rank > POPULAR_DEST_THRESHOLD) continue;
        const colonIdx = key.lastIndexOf(':');
        const destName = key.substring(0, colonIdx);
        const country = key.substring(colonIdx + 1);
        if (destName.length <= q.length) continue;  // Skip if query IS the full name
        if (destName.startsWith(q)) {
            matches.push({ name: destName, country, rank });
        }
    }
    matches.sort((a, b) => a.rank - b.rank);
    return matches.slice(0, MAX_SUPPLEMENTARY_CALLS);
}
```

#### Flow

1. `findPopularPrefixMatches(query)` scans `DESTINATION_POPULARITY_RANK` for entries within rank <= 200 whose name starts with the normalized query.
2. Returns up to 3 matches, sorted by rank (e.g., `"bar"` → `["barcelona", "bari", "barranquilla"]`).
3. Fires the primary API call + up to 3 supplementary calls via `Promise.all` (**no added latency**).
4. Merges and deduplicates results — primary results take precedence by `id`.

**Conditions**: Only activates for `type === 'any'` or `type === 'tour_region'`.

### 3.2 Transfer Hotel Brand Search

**Problem**: Brand-only queries like `"Hilton"` return empty results because the API needs location context.

**Solution**: For transfer hotel inputs, read the **paired input field** (pickup <-> dropoff) to extract a city name, then fire a supplementary search with location context.

#### Flow

1. Detect transfer inputs by ID (`tr-pickup` / `tr-dropoff`).
2. Read the paired input's value and extract city name via `extractCityName()` (text before first comma/parenthesis).
3. **Guard**: Only fire supplementary call if city name exists AND query doesn't already include it.
4. Fire two parallel calls: `"Hilton"` + `"Hilton London"`.
5. Merge/deduplicate — primary results take precedence.

```javascript
if (type === 'hotel' && (input.id === 'tr-dropoff' || input.id === 'tr-pickup')) {
    const otherId = input.id === 'tr-pickup' ? 'tr-dropoff' : 'tr-pickup';
    const otherInput = document.getElementById(otherId);
    const otherValue = otherInput ? otherInput.value : '';
    const cityName = otherValue ? extractCityName(otherValue) : '';

    if (cityName && !query.toLowerCase().includes(cityName.toLowerCase())) {
        const [primaryResults, supplementaryResults] = await Promise.all([
            fetchLocations(query, type, countryFilter, placeIdFilter),
            fetchLocations(query + ' ' + cityName, type, countryFilter, null)
                .catch(() => [])
        ]);
        const seenIds = new Set();
        results = [];
        for (const r of primaryResults) {
            if (!seenIds.has(r.id)) { seenIds.add(r.id); results.push(r); }
        }
        for (const r of supplementaryResults) {
            if (!seenIds.has(r.id)) { seenIds.add(r.id); results.push(r); }
        }
    } else {
        results = await fetchLocations(query, type, countryFilter, placeIdFilter);
    }
}
```

## Layer 4: Supporting Infrastructure

### 4.1 Text Normalization Helpers

```javascript
// Strip Unicode diacritics, lowercase, trim
function normalizeDestName(name) {
    return name.normalize('NFD').replace(/[\u0300-\u036f]/g, '').toLowerCase().trim();
}

// Capitalize first letter of each word (for API queries)
function titleCaseDestName(name) {
    return name.replace(/\b\w/g, c => c.toUpperCase());
}

// Extract city from display string: "London, United Kingdom (LHR)" → "London"
function extractCityName(value) {
    return value.split(/[,(]/)[0].trim();
}
```

### 4.2 Autocomplete Input Handler

| Setting | Value | Purpose |
|---|---|---|
| Minimum characters | 3 | No search below 3 chars |
| Debounce delay | 400ms | Prevents excessive API calls |
| Generation counter | `requestGeneration++` | Prevents stale results from displaying |

**Render pipeline**: `fetchLocations()` → `groupResults()` → `renderGroupedResults()`

## Build Script: `scripts/generate-destination-ranks.js`

Generates the `DESTINATION_POPULARITY_RANK` data from external sources.

### Data Sources

1. **GeoNames** `cities15000.txt` — population data for ~25,000 cities worldwide
2. **Wikidata SPARQL** — sitelinks count per city (proxy for global tourism awareness)
3. **Tourism overrides** (`scripts/tourism-overrides.json`) — 91 manually-curated small destinations with outsized tourism significance (Santorini, Maui, Bruges, etc.)

### Scoring Formula

- **Composite score** = weighted blend of population percentile + sitelinks percentile
- **Tourism-override entries** receive:
  - Population percentile floor of 50%
  - Sitelinks bonus of +10
- **Safety net**: For top-1,000 cities and tourism entries with <=5 sitelinks, the script estimates sitelinks from population data (Wikidata GeoNames ID mapping via P1566 is incomplete)

### Usage

```bash
# Full run (fetches from Wikidata)
node scripts/generate-destination-ranks.js

# Development (uses cached Wikidata data)
node scripts/generate-destination-ranks.js --use-cache
```

## Recommended Implementation Order

1. **Add the 4 static data structures** — airport rank, destination rank, country rank, name aliases
2. **Implement `getDestinationPopularityScore()`** — the cascading lookup scoring function
3. **Replace sort comparators** — in all grouping functions (`groupAirportResults`, `groupHotelDestResults`, `groupTourResults`, `groupTransferHotelResults`)
4. **Add `findPopularPrefixMatches()`** — and wire the parallel supplementary calls into the autocomplete handler for `any` and `tour_region` types
5. **Add transfer brand-search logic** — supplementary city-context call for hotel inputs
6. **Set up the build pipeline** — for regenerating destination ranks from GeoNames + Wikidata
